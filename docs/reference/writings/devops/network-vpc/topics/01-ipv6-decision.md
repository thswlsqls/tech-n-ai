# 01. 왜 우리는 IPv6 듀얼스택을 '아직' 적용하지 않았나

> 1차 소스: [`02-network-vpc.md` §0 — IPv6 듀얼스택 적용 여부 결정](../../../../../devops/results/02-network-vpc.md)

## 한줄 요약(Hook)

> ALB가 외부 종단을 맡는 아키텍처라면, IPv6 듀얼스택은 모던함의 상징이 아니라 SG/NACL/라우팅 테이블 규칙을 두 배로 들고 다니는 운영 부채일 수 있다.

## 핵심 질문

- IPv6 듀얼스택을 도입할 때 실제로 두 배가 되는 운영 항목은 무엇인가?
- 외부 종단이 ALB(또는 CloudFront)인 아키텍처에서 IPv6 엔드포인트의 가치는 어디서 오는가?
- "지금은 안 한다"는 결정을 어떻게 **나중에 도입할 여지**와 함께 남겨둘 수 있는가?

## 글 아웃라인

1. **들어가며 — "IPv6 안 쓰면 뒤처진 거 아닌가요?"라는 질문**
   - 듀얼스택은 가치 + 비용의 합계이며, 본 아키텍처에서는 합계가 음(-)임을 미리 던지기
2. **본 서비스의 외부 종단 구조 — 왜 IPv6 엔드포인트가 비어 있어도 괜찮은가**
   - Internet → CloudFront(WAF) → ALB → Private 워크로드
   - ALB는 CloudFront의 오리진이라 IPv6 엔드포인트가 외부 사용자에게 직접 보이지 않음
3. **두 배가 되는 운영 항목 — 회계로 비용 환산하기**
   - SG 규칙: IPv4 + IPv6 별도 작성 → 룰 한도(SG당 60개) 압박
   - NACL: 모든 인바운드/아웃바운드 룰 IPv6 미러링
   - 라우팅 테이블: `::/0` → Egress-only IGW 별도 경로
   - Flow Logs 필드 검증, IaC 모듈 양분(`ipv6_cidr_block`, `assign_ipv6_address_on_creation`)
4. **AWS 공식 권고와 서비스별 한계 — 모든 데이터 서비스가 준비된 건 아니다**
   - Fargate Platform 1.4의 듀얼스택 GA, 그러나 MSK/Aurora Writer 엔드포인트의 IPv6 한계
5. **나중에 도입할 여지를 남기는 설계 — 결정의 가역성**
   - CIDR 계획에서 Reserved 영역(`10.30.128.0/17`) 확보의 의미
   - Egress-only IGW + 서브넷별 IPv6 /64 점진 도입 시나리오
6. **결론 — '안 쓴다'가 아니라 '아직 쓰지 않는다'**
