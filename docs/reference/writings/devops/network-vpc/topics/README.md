## 단편과 시리즈의 관계 — "빌딩블록 × 메타 글" 모델

본 자료는 **단편 = 빌딩블록 / 시리즈 = 메타 글**의 상보 구조로 구성된다.

- **단편**은 02 문서를 **의사결정 단위**로 슬라이스한다. 한 편에 한 결정. 독립 출판이 가능하다.
- **시리즈**는 02 문서를 **하나의 서사**로 묶는다. 시리즈는 단편을 **인용·확장**하며, 단편이 답한 "왜"를 반복하지 않는다. 대신 다음 두 가지만 더한다:
  - **S1 (온프레→VPC 번역)** — 운영 모델 매핑 표 + 매핑이 깨지는 지점 + 마이그레이션 시나리오
  - **S2 (설계 결정 다이어리)** — 결정들 사이의 인과 사슬 + 단편 단위로는 보이지 않는 누적 효과

작성 순서는 **단편 → 시리즈**를 권장한다. 시리즈는 단편의 결론을 빌딩블록으로 사용하기 때문이다.

## 3. 폐기·병합 로그(투명성)

- ❌ **"VPC란 무엇인가 — AWS 네트워크 입문"** — 02 문서의 의사결정 콘텐츠를 활용하지 못함. AWS 공식 문서를 베끼는 글이 되기 쉬움. (관점 0/3)
- ❌ **"VPC Flow Logs 13개월 보존 — 단순 로깅이 아니다"** — 단독 글로는 02의 §3.2가 짧아 1차 소스가 얕음. **시리즈 S2 3편의 환경 분기 결정 표에 흡수**(prod 1분 / dev 10분 Aggregation Interval 항목).
- 🔁 **"GuardDuty / Network Firewall / WAF 통합 보안 레이어"** — 02의 §3.3~3.5가 각각 짧고, 보안 주제는 **06-security-and-iam.md를 1차 소스로 별도 카테고리에서 다루는 것이 적합**. 본 카테고리에서는 폐기.
- 🔁 **"Cloud Map vs Spring Cloud Discovery"** — 네트워크보다는 서비스 디스커버리/애플리케이션 토픽. 본 카테고리에서는 단편 07이 Cloud Map의 "ECS 라이프사이클 결합" 측면만 다루고, 애플리케이션 측 비교는 백엔드 카테고리로 분리.

## 작성 가이드

- **인용 정책**: 02 문서 §번호와 02 문서 하단 "6. 참고 자료(공식 출처)"의 링크만 사용. 블로그·AI 생성 콘텐츠 인용 금지(`tech-n-ai-backend/CLAUDE.md` 외부 자료 참조 원칙).
- **본문 언어**: 한국어. 고유명사·기술 용어는 영문 유지(NAT Gateway, Security Group, PrivateLink 등).
- **숫자·산술**: 02 문서에 명시된 수치만 사용. 추정값은 "추정:" 표기.
- **다이어그램**: Mermaid 권장(02 §4 토폴로지 다이어그램과 시각 일관성 유지).
- **단편 작성 시**: 글 도입부에 02 §번호를 박스로 박는다. 글 마지막의 "시리즈 인용 관계" 섹션은 그대로 유지하여 시리즈 작성자에게 빌딩블록 신호를 남긴다.
- **시리즈 작성 시**: 매 편 도입부에 "**인용 단편 + 1단락 요약**"을 박스로 두어 단편 미독자도 따라올 수 있게 한다. 단편이 이미 답한 "왜"를 반복하지 않는다.
