# 시리즈 S2. tech-n-ai의 VPC 설계 결정 다이어리 — 전 3편

> 1차 소스: [`02-network-vpc.md`](../../../../../devops/results/02-network-vpc.md) 전 절
> 본 시리즈는 **메타 글**이다. 각 편은 대응 단편 글이 이미 답한 단일 결정을 반복하지 않고, 결정**들** 사이의 **인과관계와 누적 효과**라는 시리즈 고유 가치만 더한다.

## 시리즈 훅

> 설계서는 결정의 결과만 보여준다. 결과로 가는 길에는 서로 의존하는 결정들이 있었다. 한 결정이 다음 결정의 선택지를 좁히고, 좁아진 선택지가 또 다음 결정의 비용을 정한다. 이 시리즈는 02-network-vpc.md의 결정들을 **인과 사슬**로 묶고, 단편 한 편으로는 보이지 않는 **누적 효과**를 보여준다.

## 시리즈가 단편 위에 더하는 것

각 편은 대응 단편이 이미 다룬 "**한 결정의 근거**"를 반복하지 않는다. 대신 다음 두 가지만 다룬다.

1. **인과 사슬** — 한 결정이 다른 결정의 선택지를 어떻게 좁혔는가
2. **누적 효과** — 여러 결정이 합쳐져 만든 시스템 수준의 속성(예: "데이터 유출 경로 0", "환경별 토폴로지 분기 가능")

이는 단편 단위로는 결코 드러나지 않는, **시리즈 형식이어야 가능한 가치**다.

## 편별 구성

### 1편 — CIDR 회계학: 한 줄의 산술이 만든 다섯 개의 후속 결정

- **인용 빌딩블록**: [단편 02 — /16에서 /20으로 4,096개 IP를 잡은 이유](./02-cidr-and-fargate-eni.md)
- **단편이 답한 것**: "왜 /20인가"의 산술 (Fargate ENI 한 개의 무게)
- **시리즈가 더하는 것 — 인과 사슬**:
  1. /16 환경별 + 10단위 이격 결정 → **TGW 라우팅 충돌 없음** 보장 → dev/beta/prod 동일 IaC 모듈 가능
  2. Private-App /20 결정 → ECS Blue/Green 동시 실행 가능 → 배포 전략의 자유도 확보
  3. MongoDB Atlas `192.168/16` 회피 결정 → Atlas Peering 폐기 → **PrivateLink 채택** (단편 04로 연결)
  4. Reserved 영역 `10.30.128.0/17` 확보 → IPv6 듀얼스택 미적용 결정의 가역성 (단편 01로 연결)
  5. Private-Data /24 + Private-TGW /26 결정 → 데이터 계층의 IP 소비 패턴 고정 → SG 매트릭스의 단순화
- **꼭지**:
  - 도입: "한 결정이 다섯 개의 후속 결정을 어떻게 묶는가"
  - 본문: 위 5가지 인과 사슬을 02 §번호로 추적
  - 결론: CIDR 결정을 마지막에 하면 안 되는 이유
- **02 근거**: §1.1.1, §1.1.2, §1.1.3, §1.2 전체
- **편 분량**: 김 (~4,500자)

### 2편 — '데이터 유출 경로 0'은 한 결정의 결과가 아니다

- **인용 빌딩블록**:
  - [단편 04 — VPC Endpoint와 PrivateLink](./04-vpc-endpoint-privatelink.md)
  - [단편 05 — Stateful(SG)과 Stateless(NACL)의 심층 방어](./05-sg-vs-nacl-defense-in-depth.md)
- **단편들이 답한 것**: 각각 "PrivateLink로 트래픽을 백본 안에 가둔다", "SG 아웃바운드를 비우고 NACL이 마지막 보루를 친다"
- **시리즈가 더하는 것 — 누적 효과**:
  - 4계층 서브넷(§1.2) + SG 아웃바운드 제거(§2.1) + NACL Private-Data 인터넷 DENY(§3.1) + VPC Endpoint(§1.4) **네 가지가 합쳐져야** 비로소 "데이터 유출 경로 0"이라는 시스템 속성이 성립한다.
  - 어느 하나라도 빠지면 어떤 시나리오에서 무엇이 새는지 — **침해 시뮬레이션 3편**으로 검증
- **꼭지**:
  - 도입: "심층 방어"라는 말의 진짜 의미는 누적 효과
  - 시뮬레이션 1: 컨테이너가 SSRF로 뚫렸다. 어디까지 갈 수 있는가?
  - 시뮬레이션 2: SG가 잘못 설정되어 outbound `0.0.0.0/0`이 열렸다. NACL이 막는 지점은?
  - 시뮬레이션 3: NACL이 우회되었다. VPC Endpoint Policy가 막는 지점은?
  - 결론: 단편 04와 05가 합쳐졌을 때만 보이는 시스템 수준 보안
- **02 근거**: §1.2.3, §1.4, §2.1, §3.1
- **편 분량**: 김 (~5,000자)

### 3편 — 환경별 비용 분기: NAT를 시작으로 본 IaC 모듈 분기 전략

- **인용 빌딩블록**: [단편 03 — AZ별 NAT Gateway 3개 vs single_nat_gateway](./03-nat-gateway-az-tradeoff.md)
- **단편이 답한 것**: NAT GW 한 컴포넌트의 환경별 회계
- **시리즈가 더하는 것 — 누적 효과**:
  - NAT GW(§1.3.2) 외에도 환경별로 분기하는 결정이 4개 더 있다:
    1. **VPC Flow Logs** Aggregation Interval — prod 1분 / dev 10분 (§3.2)
    2. **Network Firewall** — prod만 적용 / dev·beta 비활성 (§3.4)
    3. **Shield Advanced/WAF** — prod ALB·CloudFront만 검토 (§3.5)
    4. **Route 53 PHZ** — 환경별 독립 Zone, VPC Association 분리 (§1.5)
  - 다섯 결정이 모두 같은 IaC 모듈에서 **변수 한두 개로 분기**된다 — 이 패턴이 가능했던 전제 조건은?
- **꼭지**:
  - 도입: NAT는 시작일 뿐이다 — 환경 분기 결정의 전체 그림
  - 5가지 환경 분기 결정의 매트릭스 (prod / dev / beta 비교 표)
  - IaC 모듈 패턴: 같은 모듈, 다른 `terraform.tfvars` — 가능했던 두 가지 전제 (단편 02의 CIDR 동일 옵셋, 단편 05의 SG ID 참조)
  - 환경별 비용·관측성·보안의 회계 결산표
- **02 근거**: §1.3.2, §1.5, §3.2, §3.4, §3.5
- **편 분량**: 보통 (~3,500자)

## 읽는 순서 권장

1편 → 2편 → 3편: 설계의 **공간(어떻게 나눴는가)** → **방어(누적 효과로 어떻게 막았는가)** → **시간/돈(환경별로 어떻게 분기했는가)** 순서.
1편 없이 2편을 읽으면 "왜 Private-Data가 따로 있는가"의 인과가 약하고, 2편 없이 3편을 읽으면 "왜 prod에만 Network Firewall을 두는가"의 회계가 추상적으로 들린다.

## 시리즈 작성 가이드(공통)

- 본문에는 추측 대신 02 문서의 **명시된 수치**만 사용한다(예: `single_nat_gateway = true`, `13개월 보존`, `/20 = 4,091 IP`).
- 매 편 도입부에 "**인용 단편 + 1단락 요약**"을 박스로 둔다(단편 미독자도 따라올 수 있게).
- 매 편 마지막에 "**결정의 가역성**"을 1단락으로 평가한다(되돌리기 비용이 얼마나 드는가).
