# 시리즈 4편: 멀티스레딩 도입 전 체크리스트와 운영 안정성 검증

> 시리즈: "클럭의 벽을 넘어서 — 게임 개발자가 풀어내는 멀티스레딩 성능 최적화"

- **핵심 논점**: 멀티스레딩은 정확한 측정 없이 도입하면 복잡성만 증가시킨다. 도입 전에 확인해야 할 핵심 지표와 트레이드오프를 이해하고, 도입 후에는 C++ 게임과 Java 웹 앱 각각에 맞는 검증 전략을 수립해야 한다.

- **예상 키워드**: 멀티스레딩 성능 지표, ThreadSanitizer 레이스 컨디션, JFR 스레드 모니터링, 멀티스레딩 트레이드오프, 멀티스레딩 도입 체크리스트

---

## 다루는 내용

### 멀티스레딩 도입 전 핵심 지표

- **CPU 사용률(CPU Utilization)**: 전체 코어 평균이 아닌 코어별 사용률을 확인. 단일 코어만 100%이고 나머지가 유휴 상태라면 병렬화의 명확한 신호. 모든 코어가 이미 높은 사용률이라면 멀티스레딩이 아닌 알고리즘 최적화가 우선.
- **병렬화 가능 비율(Parallelizable Fraction)**: 프로파일링을 통해 전체 실행 시간에서 독립적으로 실행 가능한 구간의 비율을 측정. Amdahl's Law에 대입하여 이론적 최대 개선폭을 미리 추정. 병렬화 가능 비율이 낮으면(예: 50% 미만) 멀티스레딩 투자 대비 효과가 미미.
- **스레드 경합률(Thread Contention Rate)**: 이미 멀티스레딩이 적용된 시스템에서, 락 획득 대기 시간이 전체 실행 시간에서 차지하는 비율. Java에서는 JFR의 `JavaMonitorWait`, `JavaMonitorEnter` 이벤트로 측정. 경합률이 높다면 스레드를 늘리는 것이 아니라 락 범위를 줄이거나 lock-free 구조로 전환하는 것이 우선.
- **I/O 대기 비율**: `vmstat`, `iostat` 등으로 I/O wait 시간 확인. I/O 대기 비율이 높은 작업이라면 비동기 I/O나 Virtual Threads가 멀티스레딩보다 효과적일 수 있음.
- **메모리 대역폭**: 메모리 바운드 작업(대규모 배열 순회 등)에서는 코어를 추가해도 메모리 버스가 병목이 되어 성능이 선형으로 늘지 않음. `perf stat`으로 LLC(Last Level Cache) 미스율 확인.

### 멀티스레딩 적용의 트레이드오프

- **복잡성 증가**: 순차 코드 대비 동시성 코드는 상태 공간이 기하급수적으로 늘어남. 코드 리뷰와 유지보수 비용 상승.
- **디버깅 난이도**: 레이스 컨디션, 데드락은 재현이 어렵고 타이밍에 의존. 디버거를 붙이면 타이밍이 변해 버그가 사라지는 "하이젠버그(Heisenbug)" 현상.
- **테스트 한계**: 단위 테스트로 동시성 버그를 잡기 어렵다. 스트레스 테스트와 정적 분석 도구에 의존해야 하는 영역이 생김.
- **의사결정 프레임워크**: "측정→병목 식별→병렬화 가능 여부 판단→프로토타입→측정" 순서를 준수. 감이 아닌 데이터 기반 의사결정.

### C++ 게임 애플리케이션의 운영 안정성 검증

- **정적/동적 분석 도구**:
  - ThreadSanitizer(TSan): Clang/GCC에 내장된 런타임 레이스 디텍터. 컴파일 시 `-fsanitize=thread` 플래그 추가. 데이터 레이스를 런타임에 탐지하여 리포트.
  - Helgrind: Valgrind 기반 스레드 오류 탐지 도구. 락 순서 위반, 데드락 가능성도 탐지. TSan보다 느리지만 더 넓은 범위를 커버.
  - Clang의 Static Analyzer: 컴파일 타임에 잠재적 동시성 문제 탐지.
- **프레임 타임 프로파일링**: 게임에서 가장 중요한 지표는 평균 FPS가 아닌 프레임 타임 일관성. 99th percentile 프레임 타임이 16.6ms(60fps 기준)를 초과하는 빈도를 추적. Unreal Insights, RAD Telemetry, Tracy Profiler 등 활용.
- **스트레스 테스트**: 최대 부하 시나리오(대규모 전투, 수천 오브젝트 물리 시뮬레이션)를 자동화하여 장시간 실행. 메모리 누수, 데드락, 성능 저하 패턴을 탐지.

### Java 웹 애플리케이션의 운영 안정성 검증

- **스레드 덤프 분석**: `jstack`, `kill -3` 또는 JMX를 통한 스레드 덤프 수집. BLOCKED 상태 스레드의 비율과 원인 분석. 데드락이 있으면 JVM이 자동으로 리포트.
- **JFR(Java Flight Recorder) / JMC(Java Mission Control)**: 프로덕션 환경에서도 낮은 오버헤드(일반적으로 1% 미만)로 상시 모니터링 가능. 스레드 상태 전이, 락 경합, GC 일시정지, I/O 대기 시간을 시계열로 분석.
- **부하 테스트**: JMeter 또는 Gatling으로 동시 사용자 수를 점진적으로 증가시키며 응답 시간, 처리량, 에러율을 측정. 스레드 풀 크기 조정의 효과를 정량적으로 검증.
- **데드락 탐지**: JVM은 `synchronized` 기반 데드락을 자동 탐지하지만, `ReentrantLock` 기반 데드락은 JFR 이벤트나 주기적 스레드 덤프로 모니터링. 프로덕션에서는 Watchdog 스레드를 두어 응답 없는 스레드를 탐지하는 패턴도 사용.

### 두 환경의 검증 전략 비교 정리

| 항목 | C++ 게임 앱 | Java 웹 앱 |
|------|-----------|-----------|
| 레이스 컨디션 탐지 | ThreadSanitizer, Helgrind | JFR 스레드 경합 이벤트, 스레드 덤프 |
| 데드락 탐지 | Helgrind, 커스텀 락 순서 검증 | JVM 내장 탐지, JMX 모니터링 |
| 성능 프로파일링 | 프레임 타임(p99), Tracy/Unreal Insights | JFR/JMC, 응답 시간 p99 |
| 부하/스트레스 테스트 | 최대 부하 시나리오 자동 재생 | JMeter/Gatling 점진적 부하 |
| 프로덕션 모니터링 | 텔레메트리 대시보드, 크래시 리포트 | APM(Datadog, Pinpoint 등), JFR 상시 수집 |
| 핵심 지표 | 프레임 타임 일관성(스터터링 방지) | 처리량(TPS)과 응답 시간(latency) |
