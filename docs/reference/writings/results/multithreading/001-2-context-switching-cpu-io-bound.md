# 시리즈 2편: 스레드는 공짜가 아니다 — 컨텍스트 스위칭과 CPU/IO 바운드의 이해

> 시리즈: "클럭의 벽을 넘어서 — 게임 개발자가 풀어내는 멀티스레딩 성능 최적화"

- **핵심 논점**: 멀티스레딩은 만능이 아니며, 스레드를 늘릴수록 컨텍스트 스위칭 비용이 증가한다. 작업의 성격(CPU 바운드 vs I/O 바운드)에 따라 최적 전략이 완전히 달라지며, 이를 구분하지 못하면 멀티스레딩이 오히려 성능을 저하시킨다.

- **예상 키워드**: 컨텍스트 스위칭 오버헤드, CPU 바운드 vs IO 바운드, Amdahl's Law 멀티스레딩, 스레드 수 결정, 캐시 미스 성능

---

## 다루는 내용

### 컨텍스트 스위칭의 비용 구조

- 컨텍스트 스위칭이란 무엇인가: OS 스케줄러가 실행 중인 스레드를 중단하고 다른 스레드로 전환하는 과정. 직접 비용과 간접 비용으로 나뉜다.
- **직접 비용**: CPU 레지스터 상태 저장/복원, 스택 포인터 전환, 커널 모드 전환에 소요되는 사이클. 일반적으로 수 마이크로초 수준이지만, 초당 수만 회 발생하면 무시할 수 없는 총량이 된다.
- **간접 비용 — 캐시 오염(Cache Pollution)**: 스레드 전환 후 새 스레드의 작업 데이터가 L1/L2 캐시에 없어 캐시 미스가 급증. 이 "캐시 워밍" 비용이 직접 비용보다 큰 경우가 많다. Intel의 성능 분석 자료에 따르면, L1 캐시 히트는 ~1ns, L3 캐시 히트는 ~10ns, 메인 메모리 접근은 ~100ns로, 캐시 미스의 누적 비용이 상당하다.
- **간접 비용 — TLB(Translation Lookaside Buffer) 플러시**: 프로세스 간 전환 시 가상 주소→물리 주소 매핑 캐시가 무효화. 스레드 간 전환(같은 프로세스 내)에서는 TLB 플러시가 발생하지 않으므로, 프로세스 기반 병렬 처리보다 스레드 기반이 이 비용에서 유리.

### CPU 바운드 작업에서의 멀티스레딩

- CPU 바운드 작업의 정의: 실행 시간의 대부분이 CPU 연산에 소요되는 작업. 행렬 연산, 이미지 처리, 암호화 연산 등.
- 최적 스레드 수: 물리 코어 수와 동일하거나 약간 적게 설정하는 것이 원칙. 코어 수를 초과하면 컨텍스트 스위칭만 증가하고 처리량은 늘지 않는다.
- Amdahl's Law를 통한 이론적 상한 계산: S = 1 / ((1 - P) + P/N). 병렬화 가능 비율(P)이 95%이고 코어가 16개라면, 이론적 최대 속도 향상은 약 10.7배. 순차 실행 구간(1 - P)이 병목이 되는 구조를 수식으로 확인.
- Gustafson's Law와의 관계: 문제 크기를 코어 수에 비례해 키울 수 있는 경우, Amdahl보다 낙관적인 예측이 가능. 게임에서 월드 크기를 늘리거나 NPC 수를 증가시키는 것이 이에 해당.

### I/O 바운드 작업에서의 멀티스레딩

- I/O 바운드 작업의 정의: 디스크 읽기/쓰기, 네트워크 요청, 데이터베이스 쿼리 등 외부 장치 응답을 기다리는 시간이 지배적인 작업.
- CPU 바운드와 결정적 차이: I/O 대기 중 CPU는 유휴 상태이므로, 코어 수를 초과하는 스레드를 생성해도 유효하다. 오히려 스레드 수가 적으면 CPU가 놀게 된다.
- 웹 서버에서의 전형적 사례: 하나의 HTTP 요청이 DB 쿼리(~5ms)와 외부 API 호출(~50ms)을 포함할 때, 요청당 1스레드 모델에서 대부분의 시간을 I/O 대기에 소비. 이 대기 시간에 다른 요청을 처리할 수 있도록 스레드를 다중화하는 것이 핵심.
- 적정 스레드 수 산정: Little's Law(L = λW)를 활용한 추정. 평균 응답 시간(W)과 목표 처리량(λ)으로 필요한 동시 스레드 수(L)를 도출.

### 스레드 수 결정의 원칙과 예외

- CPU 바운드: N_threads ≈ N_cores (하이퍼스레딩 환경에서는 논리 코어 수까지 시도해 볼 수 있으나, 워크로드에 따라 오히려 역효과 가능)
- I/O 바운드: N_threads = N_cores × (1 + W/C), 여기서 W는 대기 시간, C는 연산 시간. Brian Goetz의 "Java Concurrency in Practice"에서 제시된 공식.
- 혼합 워크로드: CPU 바운드 풀과 I/O 바운드 풀을 분리하는 것이 실무적 접근. 하나의 스레드 풀로 통합하면 CPU 바운드 작업이 I/O 바운드 스레드를 고갈시키는 상황 발생 가능.
